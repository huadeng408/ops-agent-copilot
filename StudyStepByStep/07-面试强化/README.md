### ✅ 标准答案与展开
**Q1：最难决策是什么？**  
标准回答：在“纯 LLM”与“规则 + LLM 混合”之间做了可用性优先取舍。  
展开：纯模型更灵活，但线上稳定性和成本不可控；混合策略在大多数高频场景更稳，并保留复杂场景扩展能力。

**Q2：10 倍流量先做什么？**  
标准回答：先守住正确性和可用性，再扩容。  
展开：第一步加限流与降级策略，确保系统不雪崩；第二步识别热路径（DB/LLM），做缓存与异步化；第三步水平扩展。

**Q3：最可能事故点？**  
标准回答：写操作审批执行链和外部依赖波动。  
展开：审批状态推进 + 外部调用失败最容易导致不一致，因此要重点建设幂等、补偿和回放能力。

## ⭐ 常见面试问题清单（36题）+ 全部标准答案

### 1) 这个项目的核心业务问题是什么？
答案：在运营场景中，把“分析与执行”统一到一个受控系统里，既要高效响应自然语言请求，又要防止高风险误操作。

### 2) 为什么选择“只读直出 + 写操作审批”模型？
答案：只读查询风险低，追求效率；写操作风险高，必须治理。分轨模型在安全与效率之间取得最优平衡。

### 3) `AgentService` 最关键职责是什么？
答案：统一编排请求生命周期，串联会话、审计、路由、工具、审批分支和最终响应。

### 4) 为什么要有 `ToolRegistry`？
答案：统一工具治理口径（执行、异常、指标、审计），避免工具逻辑散落导致维护成本失控。

### 5) `MessageRouter` 与 planner 的边界是什么？
答案：高确定性高频意图优先 router；复杂长尾意图交给 planner；planner 失败可降级 router。

### 6) 如何保证写操作不能绕过审批？
答案：写工具只产 proposal，真正执行在审批服务，且审批接口附带权限与状态机校验。

### 7) proposal 幂等键如何设计？
答案：以业务意图维度构建（动作、目标、payload、发起上下文），而非请求 ID。

### 8) 审批状态机有哪些核心状态？
答案：`pending`、`approved`、`executed`、`rejected`、`execution_failed`，并有严格合法迁移规则。

### 9) 并发 approve 如何避免重复执行？
答案：状态机校验 + 乐观并发冲突检测（版本/flush 异常）+ 幂等执行判断。

### 10) 为什么保留 `execution_failed`？
答案：区分“审批拒绝”与“审批通过但执行失败”，便于补偿和故障定位。

### 11) 为什么记录 `ticket_actions`？
答案：保留动作历史和责任链，支撑审计、复盘与争议处理。

### 12) 审计日志和普通日志区别？
答案：审计日志是结构化业务事实，强调可回放和可追责；普通日志偏技术排障。

### 13) trace_id 与 session_id 分工？
答案：trace_id 表示单次请求链路，session_id 表示会话上下文，两者维度不同。

### 14) 为什么有 `tool_call_logs`？
答案：把工具执行细节（输入、输出、耗时、错误）结构化，便于评测与定位。

### 15) SQL 安全怎么做？
答案：SQLGuard + 权限限制 + 结果规模限制 + 最小权限账号，多层防护。

### 16) 为什么限制结果集大小？
答案：防止大查询拖垮系统，避免无效数据搬运，保护时延与稳定性。

### 17) LLM 超时为什么不能直接失败？
答案：业务系统可用性优先，必须有降级路径（heuristic）以保证核心服务持续。

### 18) heuristic 命中率低怎么办？
答案：增加规则覆盖、做规则冲突治理、引入离线回归数据集持续迭代。

### 19) `run_eval` 指标含义？
答案：反映工具选择准确性、任务成功率、危险动作拦截效果和延迟基线。

### 20) live 与 offline 指标为何不同？
答案：offline 只评路由；live 受模型、数据库、服务时序和异常分支影响更真实。

### 21) 压测看哪些核心指标？
答案：吞吐、P95/P99、错误率、业务状态分布（completed/approval_required）。

### 22) P95 上升但错误率不变，怎么排查？
答案：先看慢路径分布，再看 DB/LLM 耗时，再看锁冲突和外部依赖波动。

### 23) 为什么选 FastAPI + SQLAlchemy？
答案：快速迭代与异步友好，适合中台场景；代价是性能上限与调优复杂度要持续治理。

### 24) repository 抽象价值和代价？
答案：价值是语义统一与可测试；代价是抽象过度会遮蔽 SQL 性能问题。

### 25) migration 与发布顺序？
答案：先做向前兼容 migration，再灰度发布应用，最后清理旧逻辑。

### 26) demo seed 的工程价值？
答案：保证功能演示、回归验证和新成员上手都可复现，降低沟通与排障成本。

### 27) 告警如何设计？
答案：围绕业务关键路径定义 SLI/SLO，告警需绑定 owner 和处理动作。

### 28) 单点瓶颈在哪里？
答案：通常在数据库热查询和外部模型调用，需要缓存、降级与异步化治理。

### 29) 如果做多租户，先改什么？
答案：先改数据模型与权限域（tenant_id 全链路），再改索引和隔离策略。

### 30) 接入 Kafka 最适合哪类事件？
答案：审批状态变更、审计事件异步分发、报表任务触发等“事件驱动型”场景。

### 31) 审批执行如何异步化？
答案：approve 后投递任务队列，worker 幂等消费，状态机表示执行中与执行结果。

### 32) 热点缓存如何做且保证一致性？
答案：读缓存+写后失效/延迟双删+版本号校验，必要时引入 singleflight 抑制击穿。

### 33) 审计数据暴增怎么办？
答案：冷热分层存储、按时间分区、归档压缩、查询索引和保留周期策略。

### 34) 接口限流与熔断怎么做？
答案：入口限流保护系统，依赖熔断保护外部调用，超时预算保护线程池。

### 35) 迁移到 Go 保留什么不变？
答案：业务边界、状态机、幂等与审计模型不变；实现语言和基础库替换。

### 36) 你最有价值的权衡是什么？
答案：选择“可用性优先的混合策略”，在稳定上线和智能能力之间找到可交付平衡点。

## ⭐ 每个模块的“面试官深挖点”与回答思路

### API 层
- 深挖点：错误码和契约演进。
- 回答思路：协议稳定性 > 内部实现自由，强调兼容和回归测试。

### 编排层
- 深挖点：函数过长与事务边界。
- 回答思路：阶段化拆分 + 明确提交时机 + 异常路径可观测。

### 决策层
- 深挖点：混合路由必要性。
- 回答思路：成本、稳定性、解释性三维度 tradeoff。

### 审批层
- 深挖点：幂等与并发。
- 回答思路：幂等键、状态机、冲突处理三层防线。

### 数据层
- 深挖点：抽象与性能矛盾。
- 回答思路：语义抽象保留，但性能路径可直达 SQL。

### 可观测层
- 深挖点：指标是否真的可行动。
- 回答思路：每条告警必须绑定 owner + runbook + 升级路径。

## ⭐ 可扩展优化方向（高并发 / 分布式）

1. 审批执行异步化：解耦前台响应与后台执行。
2. Outbox 事件总线：审批事件可靠发布，解耦下游消费。
3. 缓存与热点治理：读缓存 + singleflight + 限流。
4. 读写分离与分区：缓解审计与分析查询对主库压力。
5. 多模型网关：按场景选择模型，控制成本并提高鲁棒性。
6. 多租户隔离：数据隔离、权限隔离、指标隔离一体化。

## 加分项C：项目亮点讲故事模板（STAR）

1. **S（背景）**：运营场景需要自然语言驱动执行，但存在高风险误操作。  
2. **T（目标）**：在不牺牲效率的前提下建立可控执行链路。  
3. **A（行动）**：实现混合决策、审批状态机、幂等控制、审计回放、可观测闭环。  
4. **R（结果）**：系统具备稳定交付与问题可定位能力，支持面向生产演进。  

按这个模板练熟，你的项目讲解会明显区别于“功能罗列型回答”。
