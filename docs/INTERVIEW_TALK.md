# 面试讲稿

## 3 分钟介绍

大家好，我做的这个项目叫 `ops-agent-copilot`，定位是一个企业运营 Copilot。它主要解决两类问题：第一类是运营数据分析，比如自然语言查询退款率、分析超 SLA 工单、生成日报周报；第二类是工单执行，比如分派工单、添加备注、升级优先级。

这个项目的核心设计点不是单纯把大模型接进来，而是把 Agent 能力放进一个可治理、可审计、可审批的后端系统里。具体来说，我用 FastAPI 做 API 层，Qwen-Agent 做 Agent runtime，SQLAlchemy 连接 MySQL，Redis 预留给缓存和会话扩展。整个系统走的是“单 Agent + 确定性工作流中间层”的架构，不是一开始就上复杂 multi-agent。

用户请求进来后，会先进入 `/api/v1/chat`。系统会生成 `trace_id`，加载最近几轮会话和摘要记忆，然后由 Agent 决定是否调用工具。只读工具可以直接执行，比如查退款率趋势、查超 SLA 工单、查工单详情、查最近发布记录。写操作不会直接落库，而是先生成 proposal，进入审批流。审批通过后，后端服务层再调用参数化写工具执行，并同步写入 `ticket_actions`、`audit_logs`、`tool_call_logs`。

为了控制风险，我做了三层治理。第一层是 SQL Guard，自由 SQL 只允许 `SELECT`，而且只能查白名单 view，必须带 `LIMIT`，不能多语句，不能访问系统库。第二层是 Verifier，会检查 SQL 安全性、结果是否为空、结果是否过大、结论是否和返回数据一致，以及写操作参数是否合法。第三层是审批流，所有危险动作 100% 进入人工审批，Agent 没有直接写库权限。

项目里还补了工程化闭环，包括 Alembic 迁移、demo 数据脚本、离线评测、审计日志、工具调用日志、失败可追踪，以及一个最简管理页，可以直接在浏览器看审批单、工单详情和审计日志。

我觉得这个项目最有价值的点，是把 Agent 从“能对话”做成了“能落地进业务系统”，重点放在工具治理、安全边界、审批链路和可观测性上，这也是我面向真实企业场景最关注的部分。

## 5 分钟深挖答案

### 1. 为什么你没有直接做 multi-agent？

我一开始刻意没有上复杂 multi-agent，而是先采用单 Agent + 中间层治理。原因有三个。

第一，当前业务边界很明确，固定就是指标查询、工单分析、报表生成、工单执行四类请求，单 Agent 完全够覆盖。

第二，真实落地里最难的其实不是 Agent 数量，而是工具治理、安全控制、审批和审计。哪怕做成 multi-agent，如果写操作不收口，风险还是很大。

第三，单 Agent 架构更容易控制复杂度，也更适合面试表达。我可以把整条链路从 chat、tool calling、verifier、approval、audit 一次讲清楚。

如果后续扩展，我会把报表生成、异常归因、写操作执行拆成不同角色，但前提是先把当前中间层治理能力打牢。

### 2. 你是怎么保证 Agent 不乱查库、不乱写库的？

我做了明确的能力隔离。

读能力方面，优先让 Agent 走业务只读工具，比如 `query_refund_metrics`、`list_sla_breached_tickets`、`get_ticket_detail`。只有业务工具覆盖不了，才允许走 `run_readonly_sql`。而这个 SQL 不是裸放的，我做了 SQL Guard，限制只能查询白名单视图，比如 `v_refund_metrics_daily`、`v_ticket_sla`、`v_ticket_detail`、`v_recent_releases`，并且必须 `SELECT + LIMIT`。

写能力方面，Agent 永远拿不到自由 SQL 写权限。它只能调用 `propose_assign_ticket`、`propose_add_ticket_comment`、`propose_escalate_ticket` 这类 proposal 工具。proposal 只是生成审批单，不会直接落库。真正执行是在审批通过后，由后端服务层调用参数化 repo 方法完成。

所以本质上我不是“相信模型不会犯错”，而是通过架构把它约束在一个安全边界里。

### 3. 审批流怎么设计的？为什么要这么做？

审批流是这个项目很关键的一环。用户发起写请求后，Agent 会先生成 action proposal，写进 `approvals` 表，状态是 `pending`。接口返回 `approval_required`，前端或者调用方再调用审批接口 approve 或 reject。

审批通过后，服务层会先再做一次 Verifier 校验，比如 ticket 是否存在、assignee 是否存在、新优先级是否合法、payload 是否完整、当前用户是否有权限。校验通过后，才真正执行写操作。

这么设计的价值在于：

- Agent 的建议和真正执行被拆成两个阶段，避免模型误操作直接改业务数据
- 所有写操作都能形成审批单，可追责、可回放
- 审批拒绝时不会产生数据库副作用

这套机制特别适合企业内部系统，因为真实环境里“可控”通常比“全自动”更重要。

### 4. 你做了哪些可观测性和审计设计？

我把 `trace_id` 作为全链路主键。每次 chat 请求都会生成一个 `trace_id`，贯穿消息、工具调用、proposal、审批、执行和返回结果。

我实现了两类日志表：

- `audit_logs`：记录业务级事件，比如 `chat_received`、`tool_called`、`proposal_created`、`approval_approved`、`write_executed`、`response_returned`
- `tool_call_logs`：记录每次工具调用的输入、输出、耗时、是否成功、错误信息

这样做有两个好处。第一，排障很快，拿到一个 `trace_id` 就能还原整条链路。第二，后续做评测和风控也有数据基础，比如危险动作拦截率、工具选择准确率、失败重试分布，都能基于这些日志继续扩展。

我还加了一个最简管理页，可以直接看审批列表、按事件类型过滤审计日志、点开工单详情弹窗，方便演示和排查。

### 5. 你怎么验证这个 Agent 真的可用，而不只是 demo？

我做了三层验证。

第一层是单测，覆盖 SQL Guard、聊天读请求、写请求必须审批、审批通过执行、拒绝审批不改库、审计日志创建、评测脚本输出等关键路径。

第二层是 demo 数据设计。我预埋了几个可演示 case，比如北京区昨天超 SLA 工单明显偏多、生鲜退款率某一天异常抬升、部分超 SLA 工单和最近发布记录时间接近，这样可以稳定演示“异常发现”和“归因分析”。

第三层是离线评测。我设计了 `eval/dataset.jsonl`，按指标查询、异常归因、工单详情、日报生成、危险写操作分桶，关注 `tool_selection_accuracy`、`task_success_rate`、`dangerous_action_interception_rate` 这些指标。尤其是危险写操作，要求 100% 进入审批流。

所以这个项目不是“模型能回一句话”就算完成，而是尽量把可运行、可测试、可评估做完整。

### 6. 如果线上继续演进，你下一步会怎么做？

我会优先做四件事。

第一，补真正的异步任务体系，比如日报生成走定时任务和消息队列，不占用在线请求。

第二，完善会话记忆和上下文压缩，让长会话下的分析更稳定。

第三，继续把权限模型做细，比如按角色限制不同写动作和不同数据范围。

第四，引入更系统的观测指标，比如 tool error rate、approval turnaround time、verifier reject distribution、agent fallback rate，把 Agent 从“能用”推进到“可运营”。

## 一句话亮点总结

这个项目最值得讲的亮点，不是我接了一个大模型，而是我把 Agent 放进了企业后端的真实约束里，围绕工具治理、审批、安全、审计和评测做了完整闭环。
